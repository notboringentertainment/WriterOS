Purpose

Decide the fate of the OpenSwarm agent integration with evidence, then consolidate WriterOS onto a single efficient agent system that delivers what OpenSwarm was originally meant to provide: agents that make tool calls and kick off workflows. The recommendation, argued below and verified in Phase 1, is to retire OpenSwarm and generalize the already-shipped Morgan runtime into the one agent kernel for the whole product. This is a surgical replacement of the agent layer (~1,700 LOC), not a repo overhaul — the composers, schemas, editor, and 1,500+ tests are healthy and stay.

Problem

WriterOS runs three parallel agent execution paths for the same product idea (the flagged "three-Morgan problem"): the Claude-native Morgan runtime behind /api/wp-chat, a raw debug bridge at /api/openswarm/writing-partner, and the OpenSwarm-backed Zoe research capability at /api/persona-capability/run. OpenSwarm is an external, separately-run Python FastAPI service on localhost:8080 that today powers exactly one shipped capability. Its settled security boundary (envelope-strong, content-limited) deliberately prevents it from calling tools inside WriterOS or mutating documents — which means the original goal, agents that make tool calls and kick off workflows, is architecturally impossible through OpenSwarm. Meanwhile the Morgan runtime already does native tool-calling with better observability (trace events, attribution guards, reach contract) and is live-verified. Every month both paths coexist adds duplicated HTTP code, a silent ops dependency (service down → Zoe research soft-fails), and split engineering attention.

Solution

One runtime, many hands. Generalize server/ai/morganRuntime/ into a persona-agnostic agentRuntime kernel: a provider-boxed tool loop where personas are configurations, capabilities are tools, and workflows are tools that invoke existing pipelines. Research becomes a native in-process tool (replacing the OpenSwarm HTTP hop) that produces the same CapabilityReceipt the UI already renders. Document composition becomes an agent-invocable composeDocument tool wrapping the existing server/compose pipeline. Document edits stay writer-gated behind the settled propose → approve → stale-guard contract. OpenSwarm, both bridges, and the external service dependency retire. Provenance gets stronger, not weaker: every capability run flows through the runtime's trace layer, feeding the Agent Observability PRD slices.

Notes

Why this is not a repo overhaul

The prompt allowed a complete overhaul. The audit says the repo does not need one: the composer pipeline (recipes, fact sheets, fidelity checks), the shared Zod contracts, the editor stack, and 141 test files are the healthiest parts of the codebase. The inefficiency is confined to the agent layer's triplicated execution paths. Overhauling healthy code to feel thorough would violate the no-deferred-debt spirit in reverse — it would create debt. The efficient system is the one that deletes a service, not the one that rewrites a product.

Rejected approaches

ApproachWhy rejected
Go all-in on OpenSwarm (move Morgan into the swarm)The settled envelope-strong/content-limited boundary exists for good reasons (provenance, voice integrity, no raw upstream in transcripts). Tool calls into WriterOS would mean tearing down that boundary or building a second trust system. Also: external Python service becomes a hard runtime dependency for the core product loop.
Keep both paths indefinitelyThe three-Morgan problem compounds: every new capability must choose a lane, every context change must be mirrored into two prompt builders, and the :8080 dependency silently degrades Zoe when absent.
Full repo rewrite around an agent frameworkNothing in the audit justifies it; 1,500+ passing tests are an asset to protect, not a legacy to escape.
Provider-agnostic tool loop (OpenAI + Anthropic)Deliberately deferred, not rejected: the boxed anthropicToolClient keeps SDK shapes in one file, so a second provider is an additive adapter later. Single-shot personas remain provider-agnostic today.

Risks & constraints

  • Anthropic-only agentic path — the tool loop is hardwired to Anthropic. Acceptable: it is one boxed file, and single-shot paths keep the AI_PROVIDER switch. Revisit only if a real second-provider need appears.
  • Streaming — anthropicToolClient uses non-streaming messages.create; long Phase 3/5 syntheses risk the known APIConnectionTimeoutError. The research tool must stream (messages.stream + finalMessage), and migrating the tool-turn client to streaming is in-scope for Phase 3 if smoke tests show latency pressure.
  • Persisted schema gravity — capabilityReceipt lives in exported project packages and localStorage. Every phase treats shared/personaCapability.ts as frozen; only the producer changes.
  • Research quality parity — OpenSwarm's Deep Research Agent may chain searches in ways a single bounded call does not. The named backend is Anthropic's web_search_20260209 server tool (dynamic filtering, supported on claude-sonnet-4-6); it allows multiple searches per call via max_uses, so the "1-2 targeted searches" contract carries over. Phase 3's parity fixture is the honest check; if quality drops, raise max_uses or grow an iterative search loop inside the runtime (it is a tool — it can loop) before any thought of reverting. Note: server tool availability differs on Bedrock/Vertex — irrelevant while the runtime targets the first-party API, flagged in case of a future provider move.
  • Dev-server staleness — every live smoke in this plan requires a dev server restart after server changes (known trap).

Future work (out of scope, forward refs when planned)

  • Observability Slices 2–5 (persisted diagnostics, cross-agent trace schema) — the runtime's trace layer is built to feed them.
  • Image/video capabilities from the original PRD Phase 3 — become runtime tools when wanted; the receipt already models them.
  • Writer's Room specialist-initiated capability calls — unlocked by the toolset matrix, gated one pair per PR.
